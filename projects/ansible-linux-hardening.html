Ansible Linux Hardening

← Back to Projects

Overview

I tightened our BTS lab’s Linux VMs so every teammate starts from the same secure baseline before we demo automation or networking work.

Problem

Manual tweaks made by different students left SSH, users, and services in unknown states. We only had a few evenings in the school lab, so rebuilding machines by hand was wasting the free access we had to the hardware.

Goal

Package baseline hardening (accounts, SSH, firewall, patching) into repeatable playbooks that can be run before every lab review and documented with evidence.

What I built

  • Inventory + host vars for Debian-based lab VMs, grouped by role (jump box vs. app nodes).
  • Reusable Ansible roles covering user policies, SSH config, package updates, and UFW templates.
  • Markdown runbook with before/after command outputs and a quick "rollback" checklist.
  • GitHub repo with tagged releases so teachers can diff what changed between semesters.

What I learned

Idempotence only shows up when you rerun the same play under pressure, so I now pair every task with a verification command before calling a system "hardened".

Architecture

A single Ansible control node targets small lab inventories; each role focuses on one baseline domain (users, SSH, packages, firewall) so we can enable/disable quickly.

Tech Stack

Automation: Ansible 2.x, ansible-lint, GitHub
Targets: Debian & Ubuntu VMs with OpenSSH + UFW/iptables

Key Features

  • Opinionated defaults for SSH ciphers, banners, idle timeouts, and sudo use.
  • Role variables for classroom vs. personal laptop inventory differences.
  • Firewall templates that only expose SSH + lab ports and log the rest.
  • Structured evidence folder with screenshots/logs for tutor reviews.

Validation / Testing

Each run goes through ansible-lint, a --check dry run, then a real apply followed by scripted SSH, user, and firewall audits so I can paste proof into tutor reports.

Multi-Cloud Storage

← Back to Projects

Overview

This capstone prototype compares how AWS, Azure, and GCP handle object storage, cost controls, and permissions when you try to keep data portable.

Problem

Labs kept locking us into one cloud, so it was hard to talk about multi-cloud strategy with evidence. I needed a practical demo that showed trade-offs, not slideware.

Goal

Design a free-tier friendly architecture that synchronizes objects across the big three clouds, enforces least-privilege access, and records cost/performance metrics.

What I built

  • Architecture diagrams + a 25-page technical report documenting each cloud’s behavior.
  • Scripts + runbooks for provisioning buckets/containers and configuring replication/sync jobs.
  • IAM policy snippets for AWS, Azure, and GCP that map to the same RBAC model.
  • Cost tracker spreadsheet + Grafana-style dashboards (screenshots) to explain spend.

What I learned

Documentation matters: the technical report became the real deliverable because it captures why some providers make lifecycle rules easy and others hide them behind APIs.

Architecture

Each provider hosts a primary bucket/container with event-driven sync to the others, centralized identity via least-privilege roles, and shared monitoring hooks for spend + health.

Tech Stack

Clouds: AWS S3, Azure Blob Storage, Google Cloud Storage
Tooling: CLI/SDK scripts, Terraform modules (in progress), Cost Explorer + Azure Cost Management

Key Features

  • Cross-cloud sync policies with documented limitations (eventual consistency, object limits).
  • Governance baseline (naming, tagging, encryption defaults) shared across providers.
  • Monitoring hooks for storage growth + alert thresholds per provider.

Validation / Testing

I run upload/read/delete tests across each cloud, measure replication delays, simulate failover by disabling one provider, and log spend deltas inside the report.
